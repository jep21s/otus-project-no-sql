# RESEARCH REPORT — Couchbase vs PostgreSQL+Redis

Микросервис заказов на Kotlin/Spring Boot с послойной архитектурой:
Controller(DTO) → konvert → Service(Model) → Persister[interface] → Repository(Entity).
Две реализации Persister: PG+Redis (Spring Data JPA + Redis, внешний кэш 60 c) и
Couchbase (Spring Data Couchbase, встроенный memory-first кэш + явный кэш 60 c).
REST: CRUD заказов/пользователей/товаров, заголовок X-User-Id.
Топологии: A (single), B (HA БД), C (HA БД + 2 app за nginx).
Протокол прогона: 3 прогона × 2 мин, перед каждым — перезапуск контейнеров, очистка и
пересидирование детерминированного датасета (1000 users, 200 products, ~10000 orders).
Нагрузка: Locust, ~1000 (базовый) / 3000 (стресс) пользователей, профили mixed/read-heavy/write-heavy.
Мониторинг: docker stats + cgroup (CPU/RAM/IO), sampling 1 Гц.

## Вердикты по гипотезам

- **H1.** Couchbase (memory-first) даёт меньшую латентность чтения «из коробки», чем PG+Redis. — _(заполняется по итогам замеров; см. таблицы выше)_
- **H2.** PG+Redis сложнее в эксплуатации, но предсказуемее по латентности записи под нагрузкой. — _(заполняется по итогам замеров; см. таблицы выше)_
- **H3.** Couchbase-кластер линейнее растёт по throughput при масштабировании, но жрёт больше RAM. — _(заполняется по итогам замеров; см. таблицы выше)_
- **H4.** При отказе узла Couchbase восстанавливается быстрее, чем связка PG+Redis. — _(заполняется по итогам замеров; см. таблицы выше)_
- **H5.** Для pet-projects (Тир A) Couchbase выгоднее по совокупному потреблению ресурсов. — _(заполняется по итогам замеров; см. таблицы выше)_
