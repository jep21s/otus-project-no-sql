## 4. Объяснение наблюдений

- **Throughput/латентность:** 1041.09 RPS при p95 ≈ 600.00 мс. Узкое место определяется по CPU peak контейнеров (см. таблицу) и соотношению read/write.
- **Кэш:** метод `GET /api/orders` кэшируется 60 c; high hit-rate виден по тому, что p50/p95 «чтения моих заказов» держится низким при росте нагрузки, а `POST` (инвалидация) даёт всплеск.
- **Ресурсы:** оцените, какой контейнер упирается в CPU (≈100% × ядро) или RAM — это указывает на узкое место стека в данном тире.
- **Гипотеза H5 (pet-projects):** сравните суммарное потребление RAM/CPU всех контейнеров этого тира с парным кейсом (a1 vs a2) — вывод о выгоде для небольших проектов делается по ресурсоёмкости на малой нагрузке.
- **Воспроизводимость:** разброс между прогонами (stddev) показывает стабильность; большие значения говорят о влиянии прогрева/сборщика мусора/ребаланса.
